# gemini_mondrian

A fourth artwork-generation route, alongside `Image_Process/mondrian/`,
`Image_Process/sketch/`, and `Image_Process/image_to_mondrian/`. This one
uses Gemini's image-to-image generation to redraw an arbitrary source photo
as a Mondrian-style **portrait** - the subject should stay recognizably a
person (same pose/proportions), just stylized with flat white/black/red/
blue/yellow color blocks and bold black grid lines, not turned into
non-representational abstract shapes - then hands that generated image to the
**existing** `image_to_mondrian` pipeline for vectorization - the
quantize/segment/fill/border-trace steps aren't reimplemented here. Output
is the same `painting_paths.json` command format every other route produces.

```text
Config profile (configs/gemini_mondrian_*.json)
  -> gemini_client.generate_styled_image()   photo -> Gemini-generated Mondrian-style image
  -> (clone downstream_template_config, point source_image.path at that image)
  -> subprocess: Image_Process/image_to_mondrian/generate_painting_paths.py
  -> output/<run>/painting_paths.json (+ preview SVG + quantized preview PNG)
```

The vectorization step runs as a **subprocess**, not an import - every other
route follows the same rule (see `webapp/route_adapters.py`'s module
docstring): `config_loader`/`path_validation`/etc. are same-named top-level
modules duplicated per route folder, so importing more than one route's code
into a single process would collide in `sys.modules`.

## Why this config is thin

`configs/gemini_mondrian_demo_a4.json` only describes the Gemini step
(`source_image.path`, `gemini.model`, `gemini.prompt`) plus which existing
`image_to_mondrian` config to clone as the vectorization template
(`downstream_template_config`). Canvas size, palette, segmentation, and
path-generation settings all come from that cloned template, not a second
copy of the whole schema here - change `downstream_template_config` to point
at a different `image_to_mondrian` profile (e.g. one with different
`morph_close_kernel_px`/`neutral_chroma_percentile` tuning) to reuse that
tuning for Gemini-sourced images too.

## Setup

```bash
pip install google-genai
export GEMINI_API_KEY=your-key-here
```

`google-genai` is this route's only new dependency, lazily imported inside
`gemini_client.generate_styled_image()` - the rest of this module (and the
rest of the repo) doesn't need it installed. The API key is read from the
`GEMINI_API_KEY` environment variable only, never written into a config file
or committed anywhere.

## Usage

```bash
python3 Image_Process/gemini_mondrian/generate_painting_paths.py \
  --config configs/gemini_mondrian_demo_a4.json
```

`--config` defaults to `configs/gemini_mondrian_demo_a4.json`. Run from the
repo root, same as every other route.

Outputs (under `output/<config-name>/`):

- `gemini_styled.png` - the Gemini-generated Mondrian-style image (debug/review artifact).
- `_gemini_config.json` - this route's own config as it was actually run
  (model, prompt, source image), saved for provenance before the next step
  potentially overwrites the file `--config` was loaded from (see "Why
  `_config.json` gets overwritten" below).
- `_config.json`, `image_to_mondrian_painting_paths.json`,
  `image_to_mondrian_path_preview.svg`, `image_to_mondrian_quantized_preview.png` -
  everything the downstream `image_to_mondrian` step produces, identical in
  shape to a normal `image_to_mondrian` run. Its `style` field reads
  `"image_to_mondrian"`, not `"gemini_mondrian"` - that's accurate, since the
  vectorization really is that pipeline running unmodified. This route's
  provenance lives in `_gemini_config.json` and `profile_name` (suffixed
  `_via_gemini`), not in the final JSON's `style` tag.

### Why `_config.json` gets overwritten

When this route is invoked through `webapp/`, the webapp writes this route's
own config to `output/<run>/_config.json` before running this script - and
this script's own downstream-config step also writes to
`output/<run>/_config.json` (so `generate_output_gallery.py`'s existing
ad-hoc-run discovery, which looks for `_config.json`, finds it with zero
code changes). That's an intentional same-path overwrite, not a bug: by the
time the downstream config gets written, this route's own config has
already been fully read into memory and copied to `_gemini_config.json`.

## Not (yet) verified

`gemini_client.py`'s exact Gemini SDK call shape (`contents=[prompt, image
part]`, reading generated bytes back out of
`response.candidates[0].content.parts[*].inline_data`) matches the
`google-genai` SDK's documented image-generation pattern, but hasn't been
run against a real API call yet - there was no `GEMINI_API_KEY` available
while building this. Config validation and the config-cloning logic
(`build_downstream_config()`) are unit tested without touching the network;
the actual API call needs a real key to confirm/fix.
